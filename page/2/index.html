blockchain_portal_noncense

目标

TARGET.isPortalActive(“orcKingdom”)

思路

  • 在合约PortalStation中，destinations["orcKingdom"] = 0xFC31cde4aCbF2b1d2996a2C7f695E850918e4007;存在于构造函数中，但没用被真正部署
  • 在函数createPortal中存在delegetecall，要求我们输入一个_destination并且返回值为true。我们知道，在合约A委托调用B合约时，执行B的逻辑，修改A的插槽。
  • 但是这里要求我们输入一个存在于destinations映射里的字符串
  • 我们可以通过Create计算出orcKingdom的地址，并将攻击合约插槽与原合约插槽对齐，并在函数connect内将isPortalActive("orcKingdom")设为true,返回值始终为true

Create 计算地址

我们知道，当depolyer和nonce值一样时，我们可以算出同样的地址

在Script环境下，存在vm.computeCreateAddress(deployer, nonce),可以预测合约地址。也就是说，如果我们设置一个循环，循环nonce == i，当返回的地址是我们所求的地址时，我们记录下当时的 i 值（targetNonce)。因为一个account初始的nonce是0，每次部署合约nonce +1。从0开始，设置一个循环,部署空合约，当nonce == targetNonce时停止，紧接着部署我们的攻击合约，此时攻击合约的地址即为所求地址

Impersonator

Impersonator

目标

证明任何人都可以打开门

思路

  • 源代码中有两个合约，合约Impersonator中的函数只有owner可以调用；ECLocker中有两个可以调用的函数

    • changeController(),当我们输入的r, v, s可以复原出初始controller的地址，controller就会被设为我们的输入的地址
    • open():当我们输入的r, v, s可以复原出controller的地址，那么门就会打开

  • 函数_isValidSignature中要求我们输入的r, v, s,不能还原出用过的签名，根据椭圆曲线算法的对称性，

    其中secp256k1 的 n 固定为FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141

  • 但是通过复原成功签名再打开门满足不了任何人都可以打开门的要求，但是如果我们输入的r v s 错误，还原的地址为0。我们可以先调用changeController()，将新的controller设为address(0)。这样，当调用open()时，我们可以输入任意数字，地址都会被还原为0，从而满足address add = _isValidSignature(v, r, s);

  • 在区块浏览器查询实例地址

    红色是真正的owner，黄色部分是找到的r, v, s

  • 最后，实例地址是合约Impersonator的地址，合约ECLocker的地址可以通过调用ECLocker()找到
