目标

成为bettor

思路

  1. 在合约BetHouse中，能使我们成为bettor的是makeBet()函数，我们需要满足两个require
    1. Pool(pool).balanceOf(msg.sender) <=20，需要msg.sender在池子中的余额大于等于20
    2. Pool(pool).depositsLocked(msg.sender)) = true
  2. 第一个require，我们初始在池子中拥有5个PDT，如果调用deposit()函数并且传入0.001ether参数为5，那么我们的_valueToMint就会变成15（合约为msg.sender铸币15）；在函数withdrawAll()中，我们发现，合约先将ether转入到msg.sender的账户中，再burn。这里就有一个重入的漏洞，如果我们在攻击合约中写一个receive函数，并在receive函数中进行第二次deposit调用，参数依旧是5（第一次调用时depositedPDT[msg.sender] += value_，调用withdrawAll()合约会向用户转账 _depositedValue = depositedPDT[msg.sender]）此时第一次调用残剩的代币尚未销毁，加上第二次调用的5，满足第一个require
  3. 第二个require，调用lockDeposits()
  4. 在脚本中授权攻击合约使用PDT的数量为5，并全部转给脚本。攻击合约attack()中授权Pool合约使用自己的全部PDT，进行第一次deposit()调用，并调用withdrrawAll()
  5. 攻击合约的receive()函数中进行第二次deposit()调用满足第一个require；调用lockDeposits()满足第二个require；最后调用makeBet()传入EOA的地址
  6. 在脚本中调用attack()函数

magic_valut

blockchain_magic_vault

目标

将mapHolder()更改为非合约地址

思路

  • mapHolder()是结构体Map的变量
  • 在函数claimContent()中，满足isUnlocked == true时，map.holder 被更改为msg.sender
  • 调用unlock()可以更改isUnlocked但是需要输入满足条件的_password
    • _password 左边八字节是owner的低八字节
    • _password 右边八字节是_secretKey的高八字节
  • secretKey通过调用_magicPassword()和generateKey()实现。虽然两个函数只能内部调用，但是我们可以复制函数逻辑我们的攻击合约中实现。参数passphrase可以使用cast storage查出来
  • 因为计算secretKey会用到block.timestamp和block.number，所以我们要保证secretKey计算与unlock()调用在同一笔交易中。如果直接在脚本中计算，vm.broadcast()会将vault.unlock(_password)作为一笔交易发送出去，但是这笔交易被打包到链上时，block.timestamp和block.number发生了变化，与我们模拟的环境不同，所以secretKey发生了变化，会出现 "Case 2 failed"的错误。
  • 所以我们应该将计算过程与vault.unlock()同放在攻击合约中，这样在脚本调用攻击合约的函数时，只会产生一笔交易，能够顺利通过检查条件
  • 使用abi.encodePacked()拼接
  • 再调用claimContent()更改map.holder
  • 成功
