测试

单元测试

单元测试是针对程序中最小的可测试单元来进行的测试，目标是验证这个单元的行为是否符合预期。

特点

粒度小：测试的是单个函数/模块

快速运行：通常不依赖数据库、网络等外部资源

目标

验证函数逻辑是否正确

检查状态变量更新是否符合预期

测试错误处理是否符合预期

集成测试

集成测试是将多个模块或组件组合起来一起测试，验证它们之间是否按预期工作。（测试多个合约间的交互是否正常）

特点

关注模块之间的协作：函数之间、合约之间、服务之间的调用

可能涉及数据库、网络、真实合约部署等外部依赖

目标

验证合约之间的调用链是否有效（vm.expectCall）

验证依赖数据是否正确流动：确保关键数据在传递过程中保持准确、一致，并未被篡改或丢失。

验证权限/角色转移是否符和业务规则

Fuzzing（模糊测试）

Fuzzing是一种自动化测试技术，主要用于发现软件中的安全漏洞、崩溃点和未处理的异常情况。它通过向程序输入随机、异常或格式化错误的数据

Fuzz测试的核心在于生成大量的随机输入，以测试智能合约在各种条件下的行为，为了帮助开发者发现边界情况和潜在安全漏洞。

特点

随机性强：自动生成大量输入组合

善于发现边界问题、崩溃、未处理的异常

目标

发现不可预见的bug

覆盖极端数值边界（0，2^256^-1)

验证合约咋让任何输入下都不会出现revert、溢出或陷入错误状态

用forge test命令运行测试

作用

  • 编译项目中的测试合约
  • 自动执行以test开头的函数
  • 输出测试结果（通过/失败）、gas用量、错误日志等信息

assertEq:是forge-std测试库提供的断言函数，使用==判断，若比较string或bytes等复杂类型，assertEq会调用哈希对比

  • 相等时，断言成功，测试继续执行或结束
  • 不相等，断言失败，Test库会触发TestAssertionFailure,本质上是revert,并标记测试为失败
    • Foundry测试框架会在终端打印对比值、文件名、行号，方便定位问他；如果在命令行加-vv，还会看到更详细的堆栈信息（v的个数越多，信息越详细）

断言函数

除了assertEq(a,b),forge-std库内还有以下断言函数

  • assertTrue(bool condition):断言表达式为真，即bool值为true时，返回成功；bool值为false时，返回失败–>revert
  • assertFalse(bool condition):断言表达式为真，即bool值为false时，返回成功；bool值为true时，返回失败–>revert
  • assertGt(a, b):断言a>b
  • assertLt(a, b):断言a<b
  • assertGe(a, b):断言a>=b
  • assertLe(a, b):断言a<=b
  • assertApprovxEqRel(a, b, maxPercent):相对误差断言，断言|(a-b)|/max(a,b)) <= maxPercent，允许的最大相对误差，按照18位定点百分比表示（1e16 = 1%, 1e14 = 0.01%, 1e18 = 100%)
  • assertApproxEqAbs(a, b, maxDelta):相对误差近似断言，即断言|a-b|<=maxDelta,当实际值于期望值绝对差不超过阈值maxDelta时，视为成功，否则revert(a为实际值，b为期望值)
  • fail():强制让合约失败（例如放在不应执行的路径中）

Cheatcodes 作弊码（测试环境控制）

来自Vm接口，用于模拟区块链环境行为，Foundry内部使用vm.*实现

是一组特殊指令，允许

  1. 控制EVM状态
  2. 伪造信息发送者（park)、签名、链上事件
  3. 读取或写入任意合约存储
  4. 调用外部命令行脚本
  5. 构造攻击场景/重现边界状态/加速测试
  • vm.prank(addr):模拟下一次msg.sender是addr(只影响下一个外部调用)
  • vm.startPrank(addr)/vm.stopPrank():持续模拟多个调用msg.sender
  • vm.deal(addr, amount):给地址addr设置余额（单位wei）
  • vm.warp(timestamp):设置block.timestamp
  • vm.roll(blockNumber):设置block.number
  • vm.expectRevert():期待下一次调用revert，否则失败
  • vm.expectRevert(bytes4):指定错误选择器的revert
  • vm.expectEmit():期待下一次emit事件
  • vm.recordLogs()+vm.getRecordedLogs:捕获日志事件，用于检查某事件是否被发出
  • vm.startBroadcast():启用该广播
  • vm.stopBroadcast():停止广播
  • vm.load(addr, solt)/ vm.store(addr, solt, value):读取或写入合约存储槽

vm.Prank()

在测试合约中引入标准库：

vm.prank(address):指定下一次调用的msg.sender

OwnerOnly指定了调用increment()的调用者必须是0x5B38Da6a701c568545dCfcB03FcB875f56beddC4

但是我们在进行测试的时候，increment()的实际调用者是OwnerOnlyTest，在这种情况下，我们可以使用vm.prank(0x5B38Da6a701c568545dCfcB03FcB875f56beddC4)指定msg.sender完成测试

也可以使用vm.Pank(address msgSender, address txOrigin)

对于多次调用，使用vm.startPark()在后续调用保持身份，使用vm.stopPark()结束它

vm.expectRevert()

vm.expectRevert()是一个预期失败的断言工具，用于确保某个操作必须失败，否则测试就失败

也可以内置revert原因（require报错的字符串）

vm.expectEmit()

vm.expectEmit()用于验证事件是否正确地被触发，包括：

  • 事件名称与参数
  • 是否使用indexed参数（即是否成为topic)
  • 发出事件的顺序，按写入的顺序对比

测试

topic1 indexed第一个参数（此处是，user)

topic2 indexed第二个参数（此处没有，默认为false）

topic3 indexed第三个参数（此处没有，默认为false）

data 非indexted参数+整个非topic（amount）的编码

我们测试时，应满足true false false true（但不会返回，vm.expectEmit()没有返回值）否则测试失败

vm.expectCall()

vm.expectCall()断言外部交互，它能验证：

  • 某个合约地址是否被调用了
  • 是否使用特定的msg.sender
  • 是否带了特定的calldata(函数选择器+参数)

不是在测试中对目标函数进行调用，而是验证目标函数有没有被其他函数外部调用
