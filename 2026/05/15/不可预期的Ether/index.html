不可预期的Ether

还是基础漏洞

  1. 违反不变量

    如果收到了意外的ETH，该判断就会错误地revert

  2. 绕过权限与限制

    该存款函数只允许白名单用户调用，但是我们使用自毁函数，可以绕过限制，直接转账。

  3. 阻止某些逻辑

    一旦address(this)的余额超过maxBalance，下面的某些程序将无法进行

常见攻击方法

selfdestruct

调用destroy()将ETH转给目标地址

用以太币预加载合约地址

“以太预加载合约地址”又名“预加载Ether到未来合约地址”

即使一个合约还没部署，我们可以提前预知它的地址，所以可以提前向它发送ETH

使用CREATE

任何人都可以通过这种方式向合约地址转账（无论它是否存在）。

==一个合约还没部署，且部署后没有receive()、fallback()，构造函数不是payable,我们也可以提前向它转ETH==

EtherGame.sol

  • 每个玩家固定投入0.5ETH
  • 有三个奖励阶段
    • 总金额达到3ETH，最后一个转入的玩家(2.5–> 3)得到2ETH
    • 总金额达到5ETH，最后一个转入的玩家(4.5–> 5)得到3ETH
    • 总金额达到10ETH，最后一个转入的玩家(9.5–> 10)得到2ETH
  • 总金额到达10ETH时开奖

漏洞点

后面三个if判断奖励阶段都使用的是currentBalance,主要判断合约余额是不是3、5、10。

  • 如果攻击者使用selfdestruct()直接转入10ETH,此时合约达到开奖标准，但玩家没有通过play()转账，导致没有获奖者，合约被锁（🤔没有人会这么做吧。。。），可以用来阻止别人获胜
  • 因为每个玩家投入固定为0.5ETH，如果攻击者使用selfdestruct()向合约转入0.1ETH,那么合约余额不再是0.5的倍数（if全为否），几个奖励阶段也将被跳过无法触发。合约被锁，造成玩家损失。

预防措施

不将address(this).balace作为检查条件

对于上述EtherGame.sol可做如下修改

这里将检查条件换成了玩家转入的金额，即使有人通过selfdestruct()向合约转账，这一部分也不会被计算到约束条件中；

同样，这部分不可预期的ether不会有任何玩家拿到，它们会被永久地锁在合约中。

ECDSA

椭圆曲线数字签名算法，是区块链中广泛使用地非对称加密算法。它是一种基于椭圆曲线密码学(ECC)的签名算法，具有密钥短、效率高、安全性强的特点。

secp256k1椭圆曲线

签名格式

以太坊签名包含(r,v,s)三部分

  • r是基于曲线随机点R的x坐标
  • s作为部署者知晓私钥的证据，是使用随机唯一nonce计算的（对应x的y坐标）
  • v是恢复签名者地址所需的恢复位，通常是27或28

但是ECDSA存在一定的安全问题(没有强制低s的情况下)

secp256k1每个x对应两个y

即每个r对应两个s

我们只要拥有一个合法签名(r,s),就可以构造另一个合法签名(r, -s mod n)进行交易

预防措施

EIP-2中强制“低s值”以防止malleability(签名可变性问题)

==签名中的s值必须小于secp256k1的阶n的一半(s<n/2)==

拒绝服务攻击（DoS）

DoS攻击是指攻击者通过合约或交易机制的漏洞，使某些功能无法执行，或者使整个合约、合约群、甚至链上关键操作失效或受阻

转账失败型

目标合约没有receive()或fallback(),转账会失败，触发revert()

King

成为King，并阻止下一个人成为King

  • cast storage slot1 ,读取prize的值
  • 传入一个>=prize的金额，成为king
  • 在攻击合约不写receive()和fallback(),或者在receive()和fallback()内部写revert(),阻止别人转账夺取king

循环遍历型

合约中通过for遍历数组执行操作，攻击者提前加入大量数据，让任何遍历都耗尽gas

register()是注册函数，rewardAll()是奖励分发函数

reward()给每个注册的用户分发1ETH,但是没有限制重复注册。攻击者多次调用register()，会耗尽合约的gas，使合约无法执行转账逻辑。最终奖金被锁死在合约中。

Pull/Push转账陷阱

PUSH模式指的是合约主动将ETH发送给用户。

限制是受益人太多时，PUSH循环发送，容易超过区块gas限制，交易回滚

transfer()/send()限制：gas限制为2300，如果接收方是合约，有复杂的逻辑，导致转账失败

如果有任何一个人地址无法接收ETH，整个奖励循环失败，所有人都拿不到奖励

易DoS

PULL模式指的是用户主动调用函数领取奖励

用户接收失败只会影响当前调用

gas 友好，只处理一个用户

抗DoS

永久状态锁定

合约限制某些i行为只能执行一次，而攻击者提前进入，使他人无法操作

攻击者提前调用，所有人永久无法参与

重入锁死DoS

攻击者在转账时进行重入调用，在回调函数中反复调用合约函数，阻断或重复操作

攻击合约

防御措施

使用ReebtrancyGuard修饰器

先修改状态再进行转账

类型转换遇到的问题

数值转换

向上类型转换前补0

向下类型转换可能会截断数据

在转换前检查范围

address和uint160转换

以太坊地址是20字节=160位.

但是address不能直接转换为uint256,需要先转化成uint160

bytes - address/uint/bytes32

bytes20 –> address

byte32 –> uint256

address –> bytes32

string - bytes32

string –> bytes

bytes –> string

string <–> bytes 不一定可逆

此时b再转换为string类型！= a

其他不可逆例子

  • abi.encodePacked()拼接多个字符串后再还原

    想从 combined 还原出 a、b 是不可能的，除非你知道边界在哪

  • Uincode字符串转换丢失信息

    通过位操作、拼接、高低位截断，可以实现多种精细控制数据结构的方式

    操作 示例 说明
    掩码截断 x&0xFFFFFFFF 保留低32位（低4字节）
    高位拼接 x<<128
    提取高位 x>>128 取出高128位
    提取低位 x&((1<<128)-1) 取出低128位

    以GateKeepOne为例

  1. require(uint32(uint64(_gateKey)) == uint16(uint64(_gateKey))表明_gatekey的低四字节与低二字节一致，即低四字节的高二字节是0

  2. require(uint32(uint64(_gateKey)) != uint64(_gateKey),要求_gatekey的低四字节不等于全部的八字节，即_gatekey的高四字节为非0

  3. require(uint32(uint64(_gateKey)) == uint16(uint160(tx.origin))

​ _gatekey的低32位（4字节）的低16位与tx.origin的低二字节一致

所以，如果tx.origin == 0xf39Fd6e51aad88F6F4ce6aB8827279cffFb92266

_gate==0x1111111100002266

tx.origin==0x78aC353a65d0d0AF48367c0A16eEE0fbBC00aC88

_gate==0x111111110000aC88
