集中流动性

UniswapV3 续

==核心思想==

LP可以选择一个价格区间$[P_a, P_b]$，只在这一个区间提供流动性，当价格离开这个区间，LP的资金会自动转换为单一资产并停止交易

==数学思想==

价格区间$[P_a, P_b]$，P是当前真实价格

$\Large x\cdot y = k = L^2$

$\Large P = \frac {y}{x}$

$\Large x = \frac {L} {\sqrt{P}}$

$\Large y = L\sqrt{P}$

xr ,yr活跃区间的实际持仓量

xv，yv来自区间边界的“虚拟、边界量”

  1. $\Large P = P_b$时

​ $\Large x = x_v + x_r = \frac {L}{\sqrt{P_b}}$

​ 因$x_r = 0$得出 $\Large x_v = \frac{L}{\sqrt{P_b}}$

  1. $\Large P = P_a$时

​ $\Large y = y_r + y_v = L\sqrt{P_a}$

​ 因$y_r = 0$得出$\Large y_v = L\sqrt{P_a}$

​ $\Large x\cdot y = L^2$ —-> $\Large (x_r+x_v)\cdot(y_r + y_v) = L^2$

==价格与原点的角度==

θ角度越大，价格越高

LP不能根据任意价格边界提供流动性，而只能在一组预定义价格点上执行

tick：价格曲线与一组预定义射线相交的位置

换句话说，UniswapV3把连续的价格区间离散化成一系列刻度，每个tick对应一个具体的价格边界，LP只能在这个价格区间添加或移除流动性

交叉的ticks不会造成干扰

尽管每段流动性曲线不连续，交易者仍然可以在报价之间无缝交易，一旦跨越一个价格变动点，UniswapV3就会重新计算剩余交易可用的流动性数量

==ticks==

集中流动性的LP可以选择价格曲线的部分来放置流动性

比如，ETH: 2000 USDC

那么多数的LP会选择在价格区间1800~2200放置流动性，ETH的价格波动较小，所以市场价在``1800~2200`波动的可能性较大，LP收取手续费的可能性更大（只有市场价在预设的价格区间内，流动性才会被启用）

价格区间允许在ticks之间波动，但不能在非刻度区间调整

代币Y相当于代币X的价格由$\Large P_y = \frac{x}{y}$得出，

报价公式

价格是这样定义的

$p(tick) = 1.0001^{tick}$

  1. tick = 0, p = 1
  2. tick = 1, p = 1.001
  3. tick = 2, p = 1.00020001
  4. tick = -1, p = 0.99990001

允许tick索引范围为 $[-887272, 887272]$

“协议不要求资产具有与价格相匹配的值“

  • 协议并不会检查或验证池子中代币的真实价格是否与曲线上计算出的价格一致
  • 它只根据V3的流动性公式和当前池子的资产数量x,y推导出一个价格

==AMM协议只关心公式定义的价格关系，而不是资产的真实市场价值。价格的一致性最终是通过时长套利者来维持的，而不是协议主动确保的==

选择1.0001作为tick的基本单位

  1. 选择相对差，而不是绝对差
    1. 使用绝对差对于不同价格水平的代币会造成巨大影响，对于一个价值100000美元的代币来说，0.1美分几乎可以忽略；但对于一个价值0.001的代币来说，0.1美分意味着价格翻了十倍
  2. UniswapV3采用了相对比例差（即百分比差异），每个tick之间的价格变化是相对价格的0.01%,不论代币的绝对价格高低，都有相同的”相对意义“

小数如何影响价格

Base网络上ETH:USDT交易池

假设真实世界价格是：

1 ETH = 1000 USDT

USDT和ETH的小数位不同

==假设1 ETH = 1000美元(人类真实价格 $P_{real}$)==

ETH有18个小数位

最小单位价值 $\Large\frac{1000}{10^{18}} = 10 ^{-15}美元$

USDT有6个小数位

最小单位价值$\Large\frac{1}{10^{-6}} = 10^{-6}美元$

尽管1 ETH 的价格远高于1 USDC，但从最小单位来说，USDC的最小单位实际上比ETH的最小单位更值钱。为了弥补小数位差异，我们需要将价格乘上一个$10^{(18-6)} = 10^{12}$的比例因子

$\Large比例因子 = 10^{decimals of token0 - decimals of token1}$

于是，$\Large P_{pool} = \frac{1000}{10^{12}} = 10^{-9}$（合约用来计算tick的价格）

$\Large P_{real} = P_{pool} \times 10^{\text{decimals}_0 - \text{decimals}_1}$

$\Large{tick} = \frac{\ln(P_{pool})}{\ln(1.0001)} \approx \frac{-20.723}{0.0001} = -207,230$

在链上执行的时候tick决定$\Large P_{pool}$

Q number Format

定点数

在solidity中，由于语言本身不支持浮点数，所以常常采用定点数(fixed-point numbers)来存储小数，核心思想是将小数乘以一个固定的“放大系数”（10^n^或者2^n^），从而把小数部分转化为整数来存储。

例如，我们无法直接存储0.5,但是可以将其乘以10^18^，这样0.5以整数的形式被间接保存下来。

Ether小数 vs Q Numbers

1 Ether = 10^18^Wei,如果想表示0.5 Ether，我们存储 $0.5\cdot 10^{10} = 500000000000000000$

这种方法能保留小数部分，但无法表示比10^-18^更小的数

==Q Numbers使用的是2的幂作为倍率==

在EVM中乘或除以2^n^可以用位移运算实现，比乘除10^n^更高效

Q Number的表示法

Q Number通常写作Qm.n形式

  • m:用于整数部分的比特数
  • n:用于表示小数部分的比特数

Qm.n总共占用m+n位(m+n个比特)，其中n是小数位

  • 整数部分可以表示的范围是 0 ~ 2^m^-1
  • 最大总值是2^m^ - 1 + (1 - 2^-n^) = 2^m^ - 2^-n^

Uniswap V3 Factory 与 Tick间距和手续费的关系

tick（价格刻度）将价格曲线离散化（即把连续的价格区间切分为离散的价格点）

tick的取值范围[-887272, 887272],因此价格曲线上有1774545个tick

tick是流动性可能发生变化的点

LP可以在两个tick之间添加流动性，但不能在非tick对应的任意价格之间添加

并非所有1774545个tick都能被某个池子使用，具体使用的tick集合取决于池在创建时的设置。

tick间距

每个池子里都定义了一个tickSpacing，用于决定相邻可用tick之间的距离

  • 如果某个池子中的tickSpacing = 10，则只有10的倍数才是合法tick
    • …-20,-10,0,10,20…
  • 如果某个池子中的tickSpacing = 60，则只有60的倍数才是合法tick
    • …-120,-60,0,60,120…

tickSpacing越大，tick越稀疏；tickSpacing越大，tick越密集

tickSpacing在池子创建时设置，并且与池子的手续费相关，每个手续费等级对应一个tick Spacing

Tick间距与波动性

UniswapV3支持不同的tick间距，以适应不同资产对的波动性

  • 高波动性对：跨越tick的交易会产生gas消耗，因此尽量减少tick跨越次数。高波动性对适合较大的tick Spacing
  • 低波动性对：价格变化不大，可以使用较小的tick Spacing,使流动性提供者能精准地布置资金

波动性与手续费

  • 高波动性资产—> 高的无常损失风险 —> 流动性提供者要求更高的手续费
  • 低波动性对（稳定币对）—> 无常损失几乎为零 —> 手续费可抵

协议只定义 tick spacing 与手续费等级的关系，由用户决定哪些资产对更稳定或更波动。

（协议不会说“这个资产对波动性大，，，”只会提供一个映射表：手续费等级 <->tick Spacing；用户需要根据资产历史价格波动、交易量、市场特性等信息，判断这个交易对是高波动性还是低波动性）

用户的选择直接影响池子参数：在创建新池子的时候，用户选择手续费等级（fee tier)，tick Spacing会根据这和手续费等级自动计算。

协议定义了规则（费率 ↔ tick spacing），用户根据资产特性判断波动性，并选择合适的费率，从而间接决定 tick spacing。

五、转换为向上舍入

将Q128.128数字转换为Q64.96数字，同时向上舍入并返回该值。

  • TickMath中存储的变量（例如tick对应的$\sqrt{P}\cdot 2^{128}$）在预计算时都是向上取整，但是tick = -1是例外
  • 计算正刻度需要ratio的倒数；但solidity中用type(uint256).max/ratio来近似；因为$\Large\frac{max value}{ratio}$无法完全精确，所以结果会略微向下舍入
  • 计算中间结果是Q128.128，结果必须转换成Q64.96,转换方式是$\Large sqrtPriceX96 = \frac{ratio}{2^{32}}$；如果$\Large \frac{ratio}{2^{32}}$不是整数，代码会加一，使最终结果向上舍入。

也就是说：

  • getSqrtRatioAtTick()的数学目标本来应该计算：

    $\sqrt{1.0001^\text{tick}} \times 2^{96}$

  • 但因为 Solidity 不能使用小数，只能用整数加移位（fixed point）

  • 所以计算过程必须：

    • 使用 近似值
    • 使用 预计算常量
    • 使用 向上 / 向下舍入
    • 使用 Q128.128 → Q64.96 的截断
    • 使用 type(uint256).max / ratio 作为倒数

UniswapV3中的真实和虚拟储备

UniswapV3有两种类型的储备金：真实储备金和虚拟储备金

每个分段（segment）都有一个对真实储备：代币X的真实储备（Xr），代币Y的真实储备（Yr）,其中一种代币的储备金可能为0，但两种代币的储备金不能同时为0，否则这部分就没有流动性

虚拟储备代表了一个阶段将拥有的代币数量，如果这个阶段延伸到无穷大，就和UniswapV2一样

实际储备

显示了单个价格区间，当前价格由红色射线表示

代币X实际储备是价格x坐标与上方报价的X坐标之间的距离，对于代币Y也一样，但现在它是y到较低刻度的距离

x_real = 当前价格与上界 tick 的水平距离（几何意义）

y_real = 当前价格与下界 tick 的竖直距离（几何意义）

UniswapV2使用的是无限曲线$x\cdot y = k$，一个segment是有限曲线，它不能无限延伸

通过swap，价格移动到下图所示的位置，在这种情况下，实际代币储备将仅由代币Y组成

另一种可能是价格向较低的刻度移动，在这种情况下，该分段的实际储备将完全由代币X组成

实际场景下的交易池由多个段组成，在这种情况下，价格可能会跌破蓝色部分进入紫色部分

当价格低于蓝色曲线中的最低价格，蓝色曲线的Y代币将不再出售，只要价格继续低于最低刻度线，蓝色部分中的X代币数量就会保持不变

每个segment都存在真实储备，即每个段都有一对值，其中一个可以为0，但不能同时为0

当价格在一个区间内变动的时侯，代币X和Y通过互换进行交易。因此，当价格在其中时，每个分段的功能类似于UniswapV2。

虚拟储备

虚拟储备是指如果某个段是无限曲线的一部分而不仅仅是一段的情况下，该段将拥有的储备。

虚拟储备就是我们在UniswapV2中的储备，

x_virtual = L / price y_virtual = L * price

这两个中不是segment 实际拥有的 token，而是假装这个segment是无限曲线时，它有多少token

随着曲线段的增长，该段的实际储量接近虚拟储量，如果曲线延伸至无穷大（如UniswapV2 中那样），虚拟储备和实际储备将变得相同

根据当前的价格和流动性计算虚拟储备

$L^{2} = x\cdot y$ $\Large p = \frac{y}{x}$

$\Large\frac{L^{2}}{p} = \frac{x\cdot y}{p}$

$\Large\frac{L^{2}}{p} = x^{2}$

$\Large\frac{L}{\sqrt{p}} = x$

$ y = L\sqrt{p}$

UniswapV3中的实际储备

段的实际储备是该段中包含的代币数量—— 代币X ,代币Y或两者

虚拟储备是该段拥有的代币数量（该段是无限曲线的一部分）。虚拟储备是必要的，因为UniswapV3中曲线段中的互换行为与UniswapV2中完全相同，也就是说，就好像这些段是UniswapV2池中无限曲线的一部分

图中可以看出，具有相同流动性的两部分，它们的虚拟储备相同，但实际储备不同

所以我们不仅能从价格和流动性中得出实际储备——还必须考虑细分边界

一个分段(segment)中的实际储备量

某个分段中X的实际储备量是在swap期间将离开该分段的代币数量，该交换使价格移至上限

某个分段中代币Y的实际储备量是在swap期间价格移动到较低价格的阶段，如下图所示

因此，如果我们可以计算第一个例子中有多少代币X离开了池子，以及第二个例子中有多少代币Y离开了池子，我们就可以得出实际储备的公式

推导代币X实际储备的公式

$x_r = x - x_u$

$x_u$是价格区间中，y的最高价格

$\Large x_r = \frac{L}{\sqrt{p}} - \frac{L}{\sqrt{p_u}}$

当$x = x_u$时，$p = p_u$

我们可以得出，x的实际储备量变为0

当价格高于价格上限时，x的实际储备量为0

$\Large x_r = \frac{L}{\sqrt{p}} - \frac{L}{\sqrt{p_u}}$

当价格低于价格下限时，y的实际储备量为0

$\Large y_r = y - y_l$

$\Large y = L\sqrt{p} - L\sqrt{p_l}$

UniswapV3中的仓位

在UniswapV2中，当LP添加流动性时，会收到池子的份额（LPT），即ERC-20流动性代币，代表他们对池子资产的占比（包括手续费）。这些LP token是可替代代币。

在UniswapV3中，LP选择一个特定的区间来提供流动性，因此协议必须以不可代替的方式逐个记录每次存入的流动性，这就引入了Position（仓位）的概念

当LP在一个区间内添加流动性时，可以认为他们在打开或修改一个仓位

  • 若该仓位不存在，则成为打开
  • 若该仓位已存在，则称为修改

UniswapV3存入流动性由mint()实现，含义变成了铸造一个仓位

仓位存储在UniswapV3Pool合约中的一个名为positions的mappings中

该position的key由以下三者通过Keccak哈希生成：

  • Positon owner的地址
  • lower tick
  • upper tick

如果一个地址0xA第一次在tick -10到tick 10之间添加流动性，那么会创建一个key

在UniswapV3中，仓位是不可替代的，因为仓位绑定了不同的价格区间，因此当两个LP赎回流动性时，他们的价格区间不同，贡献的真实储备不同，他们可能得到完全不同的代币数量

仓位可以通过外围合约包装成ERC-721（NFT），例如官方的 NonfungiblePositionManager，但核心合约本身不是NFT，也无法转移仓位

核心合约只负责：

  • 打开仓位
  • 修改仓位
  • 移除流动性

UniswapV3的手续费

在V3中，仓位只会获得它区间内的swap所产生的手续费

协议用以下自段来帮助计算仓位可领取多少手续费：

  • feeGrowthInside0LastX128
  • feeGrowthInside1LastX128

原理

  • 协议记录池子自创建以来累计的手续费
  • 记录每个tick上方与下方累计的手续费
  • 从而推算某仓位区间内累计的手续费
  • 再结合仓位自身的lastX128字段得出该仓位应得多少手续费

mint()

  • recipient：仓位拥有者
  • tickLower、tickUpper：价格区间
  • amount：流动性（非负），用于表示想添加多少流动性
  • data：用于回调(非空时开始闪电贷)

所以，调用者必须实现回调，EOA无法实现回调函数，EOA调用mint一定会失败

因此LP必须通过一个中间合约：Position Manager（例如官方的 NonfungiblePositionManager）

流程如下：

EOA –> PositionManager –> core pool mint –> 回调 –> PositionManager 转账 –>完成mint

PositionManager负责：

  • 接收用户指定的token数量
  • 将token数量转换为liquidity
  • 再调用core mint
