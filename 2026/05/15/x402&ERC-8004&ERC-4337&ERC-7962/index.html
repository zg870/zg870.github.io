HTTP 402（Payment Required）

用户必须先完成支付，服务器才会返回请求。

但是在传统的Web中，因为浏览器没有统一的支付协议、Web支付体系不统一、网站一般用的是登录+订阅系统来代替，它几乎没有被真正使用

随着AI Agent 自动调用API的出现，HTTP 402有了真正的应用场景

AI Agent：能自主感知环境、决策并执行任务的AI程序

与普通AI只能回答问题、回应单次请求并且不会调用工具对比，AI Agent 能够自动完成任务，保持持续执行，且会调用API/工具

以前的支付系统不适合机器，AI Agent 无法登录、输入验证码、处理网页，所以需要机器可读的支付协议

目前最典型的实现思路是： ==HTTP 402 + 加密货币 + AI Agent + API网关==

很多项目把这一套体系称为 x402 protocol

x402：是一个开放的HTTP协议，核心是实现机器自主微支付、链上即时结算，是Web和Web3的价值连接器，兼容多链与ERC-2612等标准，主打稳定币支付

核心思想：利用HTTP 402状态码，使用稳定币支付，通过区块链验证支付，支付成功后服务器返回资源

解决了什么

  • 机器/AI/Agent支付难题：实现Agent自主结算，支持小额微支付，摆脱人工干预与预充值束缚
  • Web3支付门槛高：用户无需持有主币支付Gas，仅需钱包签名，开发者易集成
  • 数字服务变现缺陷：替代订阅制/广告模式，实现按使用收费，降低变现摩擦
  • 跨系统信任成本高：基于区块链实现可追溯，防篡改，降低跨生态支付开发成本

实现细节

  • 目前主要支付USDC，价格稳定，微支付不会波动
  • gasless支付，用户只是签名，relayer提交交易并支付gas（也就是第三方、服务中继、服务器）。
  • facilitator(支付中继，支付验证服务器)，用于验证支付、提交链上交易、返回支付状态

特点

  • 支付直接集成在HTTP协议中

  • 机器对机器支付

  • 微支付，实现$0.0001 $0.001 $0.01级别的API收费

  • 无账户模式，不需要登录、API key 、订阅，支付就是身份

  • 链无关（是协议不绑定区块链，不是跨链交易），x402本身并不依赖某一条链，它只是规定了一种HTTP支付格式

x402使用区块链的优势

  1. 去中心化的支付验证，传统系统中，支付需要依赖第三方机构。使用区块链的优势是：签名+链上验证 API 可以安全地验证支付，不依赖中心化网关
  2. 防止重放攻击/跨链防篡改，签名里包含chainId nonce deadline,即便AI Agent或者第三方尝试多次使用同一签名，链上和服务器规则会阻止重复支付
  3. 用户不用自己支付手续费
  4. 跨平台和链无关，任何链上验证者都能核实签名和资产状态，传统支付系统无法在不同平台、不同资产间做到同样程度的统一验证

x402核心解决的是**“如何完成支付”的问题，与之互补的ERC8004则是专注解决“支付主体是谁”以及“为何可以信任支付方”**的信任与身份验证问题

ERC8004：以太坊上“AI智能代理”的无信任协作标准，基于ERC-721，让链上AI代理可以被唯一标识、可信评估、安全协作。在AI经济中，x402负责处理高频微额的支付，而ERC8004负责验证对方的合法性，防止恶意机器人滥用资源。

ERC8004的完整工作流程

ERC-4337：以太坊上的账户抽象（Account Abstraction）标准。它允许用户通过智能合约钱包取代传统外部账户（EOA），在无需更改共识层协议的前提下实现更灵活、安全的账户模型。

外部账号EOA和智能合约账户SCA都仅限于使用以太坊唯一的签名算法，即椭圆曲线算法（ECDSA）。两种账户必须以区块链的原声代币支付gas费用

ERC-4337是一套通过设立独立的“替代内存池”来实现账户抽象的方案。即账户充当智能合约的钱包，用户可以在不拥有自己的私钥或无需维护以太币已支付交易费用的情况下与以太坊网络进行交互。它使用替代内存池(Alt Mempool)设计接收和解释消息。

传统交易

ERC-4337

特点

  • 多操作：与智能合约的某些交互涉及多个步骤，当与DeFi协议交互时，用户可能需要批准代币转移，然后存入代币，再执行其他操作。每个步骤都需要EOA账户单独发起。ERC-4337引入了”UserOperation”新交易类型来解决这个问题。它允许用户定义应作为单个操作的一部分执行一系列步骤，然后这些步骤由EntryPoint合约执行

  • 多签名：传统EOA是一个私钥对应一个账户，如果私钥丢失，那么资金永久丢失。但是ERC-4337钱包是合约，可以自定义验证逻辑，例如：

    ==2/3 multising==区块链里的多签名规则，意思是三个授权者至少两个签名交易才会执行。

    也就是说，在单个私钥泄露时，ERC-4337的多签名规则可以保障账户资金安全。

    多签名还能做一个很重要的功能——社交恢复。当一个私钥丢失时，2/3可以恢复账户

  • 自定义签名方案：传统以太坊交易使用的签名方式是ECDSA，节点在进入区块前会自动验证签名。但ERC-4337的关键设计是EntryPoint不验证签名，而是调用用户账户中的validateUserOp()，具体的签名逻辑完全由钱包合约自己实现，因此钱包可以自由定义签名规则。换句话说，在ERC-4337中，签名验证从协议层下发到了账户合约层，通过wallet.valudateUserOp()来决定一笔UserOperation是否有效，从而实现高度可定制的账户安全授权机制。

  • Gas成本灵活：ERC-4337支持”赞助交易”即其他人可以支付Gas费用，从而在交易费用处理方面提供 更大的灵活性

  • 可升级性：ERC-4337标准使用代理合约，使其可以“升级”。允许在不中断生态系统内互操作性的情况下添加新功能和修复错误

UserOperation是ERC-4337的交易，结构类似

相当于user提供给以太坊账户的待办事项列表，可以包括转账、与智能合约交互，甚至是多个操作的组合

UserOperation Mempool用户不是将交易发送到常规以太坊内存池，而是将UserOperation对象发送到称为Alt-Mempool的单独内存池

Bundler（捆绑器）是在ERC-4337中负责收集用户提交的UserOperation的节点，它会从UserOperation Mempool中获取多个用户操作，对其进行模拟验证并根据Ges fee等激励机制进行优先级排序，然后将这些操作打包成一次线上交易。

Bundler类似于权益证明（PoS）网络中的验证者，都会从待处理池中选择交易、按费用排序并进行打包，只不过验证者打包的是普通交易，而Bundler打包的是UserOperation

Global Entry Point全局入口点，Bundler在收集多个UserOperation后，会把他们打包并发送到该合约，通过调用handleOps()统一处理。Entry Point会依次调用各个账户的validateUserOp()进行验证，并在验证通过后执行对应得操作。同时负责Gas结算、Paymaster支付等逻辑。

Contract Account合约账户，账户抽象体系中的用户账户实现，它是一个只能合约而不是传统的外部账户（EOA）。与需要用户手动发起交易的EOA不同，它可以根据接收到的UserOperation自动执行逻辑，例如验证签名、调用其他合约或按照预设规则做出决策，因此能够实现多签、社交恢复、自动化交易等复杂功能

Paymaster是一个可选的智能合约，用于为用户的UserOperation支付或赞助Gas费用 。它会在UserOperation验证阶段参与流程，并向Builder报销交易执行所消耗的Gas。它的资助规则都可以通过智能合约逻辑自定义实现

Aggregator也是一个可选的智能合约，用于优化和聚合多个UserOperation的签名验证过程。它可以将多个用户操作的签名合并或统一验证，从而减少链上验证成本，提高执行效率，尤其适用于需要批量签名验证的场景，例如多签钱包或特定签名算法体系

一次完整的ERC-4337的交易流程

ERC-4337通过可编程账户、灵活签名和Gas抽象，让区块链交易更自动化、更易用。

ERC-4337为AI Agent提供了可编程的钱包和自动交易的能力，ERC-4337的Contract Account可以根据预设逻辑自动验证和执行UserOperation，AI Agent可以自主构造并提交操作来完成链上交易。同时，Paymaster可以为交易代付Gas，使AI或应用无需持有ETH也能执行操作，而自定义签名和批量执行能力则允许AI实现多签决策、自动投资、自动支付等复杂行为。

ERC-4337让AI Agent能够安全地管理资产并自动参与区块链经济活动

ERC-7962：Key Hash Based Tokens基于密钥哈希的token

在ERC-20和ERC-721中，所有资产都是用token -> address来表示的，如mapping(address => uint256) balances，也就是说资产直接绑定到Ethereum地址。

这种设计存在几个问题：

  • 地址隐私性差，所有的交易都是公开的，链上分析工具可以追踪资金流、识别用户、关联多个地址
  • 地址绑定性太强，如果私钥泄露，用户必须用transfer将所有tokes转出，才能迁移财产
  • 不支持Relayer/Gas Sponsor,必须由持有地址私钥的人发送交易

因此ERC-7962提出“用公钥哈希（keyHash）代替address表示所有权

ERC-7962的关键改变是token -> keyHash,keyHash = keccak256(publickey),链上只存储keyHach不存储地址, owner = keyHash

（Ethereum的地址本质是address = last20bytes(keccak256(publicKey))）

ERC-7962的交易通常包含签名授权结构

交易的执行流程：

  • 资产拥有者签名，例如transfer(keyHash_from, keyHash_to, amount, nonce)
  • Relayer提交交易，任何地址都可以提交
  • 合约验证签名，合约执行recover(publicKey, signature),然后验证keccak256(publicKey) == keyHash，如果成立，transfer success

关键特点：交易提交者 ！= 资产拥有者

该提案定义了两个主要接口

IERCKeyHash721:对于非同质化代币（NFT），每个代币都有唯一的tokenId标识，所有权通过keyHash(keccak256(key))进行管理

IERCKeyHash20:对于同质化代币，其余额与keyHash关联

HTTP 402 + Web3支付架构工作流程

当用户向AI Agent发送请求时，AI在执行任务过程中可能需要调用各种API（数据查询、计算服务等），这些API 往往采用小额高频收费模式。在传统模式下，每次调用API时，AI都需要将支付页面推送给用户，用户手动支付完成。不仅极大降低工作效率，同时在执行多笔交易时，会耗费大量gas费

而在 HTTP 402 + Web3支付架构下，支付流程可以完全自动化。用户只需要在开始时通过钱包进行一次授权签名，之后AI Agent就可以在授权范围内自动完成支付。

  1. 用户授权：用户的钱包弹窗进行一次签名授权。由于使用ERC-4337账户抽象钱包，这个签名可以一次性授权多笔交易，而不需要每次调用API都进行签名

  2. 签名结构：授权签名遵循ERC-7962的设计：

    1. nonce：限制签名可使用的次数
    2. deadline：限制签名的有效时间
    3. 防止签名被重复使用

  3. 自动支付：当AI Agent调用收费API时：

    1. AI将签名附在请求中
    2. API服务验证签名后即可完成支付

    这一机制结合HTTP 402 状态码实现自动结算

  4. Gas费用问题：在ERC-7962设计中引入 Gas Sponsor/ Relayer：

    1. AI的交易不需要用户支付Gas
    2. 中继者代为支付交易gas
    3. 用户只承担实际API费用

  5. 身份验证：AI Agent的身份通过ERC-8004进行验证，确保：

    1. Agent的身份可信
    2. Agent的权限明确
    3. Agent可以被链上验证

  6. 任务执行与结果记录：AI Agent完成任务后：

    1. 生成执行报告
    2. 将执行结果上链
    3. 用户可以验证：
      1. 命令是否执行成功
      2. Agent是否授权使用资金

简单来说就是，在Web4架构中，AI Agent通过ERC-8004建立链上身份与信誉系统，用户利用ERC-4337抽象账户授权，并通过ERC-7962生成可限制次数和时间的签名，使AI能在HTTP 402支付模型下自动调用收费API；Gas由中继者赞助，任务执行结果上链并持续更新Agent信誉，从而形成一个可自动支付、可验证、可追责的AI Agent经济体系。
