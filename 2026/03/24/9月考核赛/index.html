Itname

目标

yourname == true

思路

  1. 在函数check()中,满足四个modifier,yourname就会被设为true

    1. level1，tx.origin != msg.sender,要求函数调用者不能是EOA，所以我们要使用一个攻击合约Attack。msg.sender 就是Attack合约，tx.origin 是我们的EOA，即可通过这个修饰器

    2. level2，require(uint256(uint160(msg.sender)) % 77 == 7);,对我们的Attack合约做出了要求。Create2具有预知合约地址的功能，可以使用Create2在工厂合约中预测Attack的地址，使uint256(uint160(address(Attack) )) % 77 == 7,就能通过level2

    3. level3,address(this).balance > 0,我们需要给目标合约转账，但是该合约中receive()函数会直接revert，我们没办法通过transfer、call、send等常规方法转账。但是，solidity中的自毁函数(selfdestruct())，可以通过合约自毁的方式给任意地址转账。虽然我们已经有了一个攻击合约Attack,但是通过level3需要在调用check()之前执行，但是自毁后我们我方再进行函数调用，所以此时我们需要一个单独的合约向它转账，再调用它的自毁函数，实现address(this).balance > 0,且不影响Attack后续函数调用

    4. level4，wishes[tx.origin] > 1 ? true : false == true，根据三元运算符的运算规则我们需要实现wishes[tx.origin] > 1 ，modifier返回true

      1. wishes是一个映射，我们可以在函数wish_making()中改变它，但是这个函数依旧有两个modifier限制。

        started[tx.origin] == true&wishes[tx.origin] > 0，但当我们调用start_challenge()时，两个modifier会被同时满足，再调用regert(),

        此时

      2. 该函数中初始化了接口Wish_Maker的实例，并设置了bool is_less_than = false;;当该实例调用Wish_Maker的函数

        1. 第一条路，如果wish_amount()返回值小于1时，is_than = true,wish_made[tx.origin] == true。然后会进入下一个if判断语句，wishes[tx.origin] = wish_maker.wish_amount
        2. 第二条路，如果wish_amount()返回值大于1时，is_than依旧为false,wishes[tx.origin]--

      3. 因为我们的要求是wishes[tx.origin] > 1 ，我们可以控制函数wish_amount第一次被调用返回为0，第二次返回为2（第一条路），那么wish[tx.origin] = 2满足level4。因为源码中Wish_Maker只是一个接口，并且wish_amount()函数中并没有任何逻辑。但是进行函数调用时，编译器只检查函数选择器，我们可以在Attack中创建一个同名同参的wish_amount()，并编写逻辑。wish_amount()是一个view类型的函数，并不能在内部写改变变量的逻辑或调用能够改变变量的函数。但是，因为我们目前选的是第一条路，在两次if判断中间，public类型的wish_made[tx.origin](有同名的getter函数)发生了变化，由true变为false,我们可以在wish_amount()逻辑中加入wish_made[tx.origin]判断。

  2. 先利用工厂合约计算一个满足level2的Attack合约，Attack内部的attack函数中要依次调用start_challenge()、regert()、regret_making() ，check()、isSolved()的逻辑;另一个函数wish_made()的逻辑如上

  3. 再写一个Hacker合约，有receive()函数接收EOA转账，函数destroy()中的selfdestruct()函数内是目标合约的地址

  4. 先向Hacker合约转账，调用destory()、attack()

  5. 成功

GreedyBlueMoon

目标

paralysisRing >= 1

思路

  1. 首先paralysisRing==0,有两个函数可以使paralysisRing增加为1

    1. collectingBossLoot(),想要调用它需要满足

      waitTime[msg.sender] < block.timestamp

      attackingType[msg.sender] == 2,attackingType[msg.sender]初始化为0，调用fightMob()可以让其更改为1，满足不了要求；调用fightBoss(),虽然可以满足attackingType[msg.sender] == 2,但是waitTime[msg.sender] = block.timestamp + 52 weeks,满足不了waitTime[msg.sender] < block.timestamp

      ==此路不通==

      1. redeemingParalyzingRing(),有三个约束条件

        1. shards[msg.sender] >= price,有两种方法可以增加shards

          1. buyShards(),需要我们转账shards[msg.sender] == msg.value /rates,rates可以读取插槽获得

            根据greedyBlueMoon的存储布局，以及solidity存储打包的原则，我们可以得到每个数据存储的插槽，使用cast storage 0xcontractaddress slot --rpc-url http:...，即可读出插槽的具体内容。如下是各个数据对应的插槽以及它们存储的内容：

            price为36666，rates为一个巨大的数值，如果我们传1 ether,仅能得到shards[msg.sender] = 0.09想要达到36666所需要的ether太多，==此路不通==

          2. newchallengerBundle() ，调用它shards[msg.sender]+=bonus(+66)它要求challenger[msg.sender] == false并且uint16(uint160(msg.sender) >> 144 >= high2Prefix,challenge[msg.sender]初始为false,但是一次调用之后变为true，说明一个地址只能薅一次。并且high2Prefix初始为0，每次函数被调用后增加64。如果我们有556个地址，每个地址都调用一次newchallengerBunfle()并调用transferOfItems()将shards转给EOA，那我们就能通过这个require。（使用Create批量生成，每次地址的高二字节+64。因为题目存在一定限制，不能一次性生成这么多地址，我们可以把盐值设为全局变量，每次生成地址使用的盐值不同，避免分多次调用工厂合约生成的地址发生碰撞）

        2. 我们需要输入一个_key1，并满足(keccak256(abi.encodePacked(password1))==keccak256(abi.encodePacked(_key1))),由于我们已经读出password1为0x727765623636000000,作为_key直接传入即可

        3. 我们需要输入一个_key2，并满足keccak256(abi.encodePacked(password_2)) ==keccak256(abi.encodePacked(_key2)))

          _password_2的计算公式

          _password_2的32字节为passwors_2_0、passwors_2_1、passwors_2_2、passwors_2_3按要求依次拼接而成

          它们被存储在数组password2中，我们可以在chisel中查它们的具体存储位置，返回的是第一个元素存储的位置，依次加1即为剩下几个元素的具体位置

          读取数值

          所以_password_2 = 0x8542365815820425982536745843621597268413585632246952148523852424,作为_key2传入即可

  2. 使用工厂合约批量生成满足条件的Attack地址，并在Attack合约中attack()函数写调用newchallengerBundle()并通过transfersOfItems()将shards传给EOA的逻辑

  3. 多次调用createaddr(),每次生成30个符合要求的合约，函数内部有调用Attack合约attack()函数的逻辑

  4. 调用redeemingParalyzingRing()，传入_key1、_key2

  5. 调用isSolved()

  6. 成功

MagicPool

目标

target.totalSupply() == 10 ether && address(target).balance < 1 ether

思路

  1. 增加totalSupply的函数有deposit()存款函数，当我们通过deposit()给目标合约转账时，我们totalSupply += msg.value。但是直接通过该函数增加totalSupply时，需要我们转10个ether给目标合约，address(target).balance = 10 ether,并不能满足通关条件

  2. 目标合约中还有一个flashLoan()函数，闪电贷函数，借款人必须在同一笔交易中归还借款。

    1. amount <= maxloanamount,要求我们一次借款最大的数额不能超过10 ether

    2. balanceBefore >= amount，即address(target).balance >= amount，我们可以使用cast balance 0xcontractAddress --rpc-url http://....查询目标地址的余额，正好是10 ether

    3. 调用FlashLoanReceiver接口中的函数excute()

    4. 最后要求address(this).balance >= balanceBefore，即要求我们在调用excute()函数后，立刻将借款还回去

      • 我们可以由excute()函数入手（在Attak合约中构造一个同名同参的函数，自己改写逻辑），如果我们在该函数内部调用deposit()那么totaSupply与msg.value被更改为10ether,balances[msg.sender]也被更改为10ether，然后我们进行还款。此时totaSupply与balances[msg.sender]并不会因为还款而减少，所以我们实际向目标合约转账为0就能满足target.totalSupply() == 10 ether

      • 如果再调用withdraw(),目标合约会给我们转balances[msg.sender]，因为我们实际并未在目标合约存钱，合约只能从它原始余额向我们转账，这样就能把合约余额掏空，满足address(target).balance < 1 ether

  3. 新建一个Attack合约，包含三个函数

    1. excute()，包含调用deposit()函数、授权目标合约地址以及向目标合约地址还款的函数
    2. attack(),调用flashLoan()函数，并将合约余额全部转给EOA
    3. receive()，接收目标合约转账

  4. 在脚本中调用attack()

  5. 调用isSolved()

xiaoheizi

目标

controlVote.Votes(msg.sender) > 100

思路

  1. 通过调用buyVotes()，并向其转账我们可以获得Votes但是一个ether,只能获得一个投票权，==此路不通==
  2. 我们可以看到在swap()函数中,Votes和shares是1:1的关系兑换的关系，而在函数vote()中，如果给别人投票自己只减少shares，不减少Votes。被投票方则是Votes与shares同时增加。如果我有两个账户都有一定的shares，就可以相互投票，免费增加Votes
  3. 调用swap()需要我们是白名单用户，加入白名单函数joinWhitelist(),要求我们不是白名单用户，且Votes[msg.sender] >= 1那就需要我们先调用buyVotes()传入1 ether,购买一个Votes。再调用swap()将Votes转换为shares
  4. 我们需要两个账户互相转账，所以两个合约都要有步骤3
  5. 在脚本中向两个合约均转账1 ether
  6. 分别调用它们的第一个函数，将两个账户都加入白名单
  7. 然后利用for循环使两个合约相互投票（循环次数要大于100）
  8. 在任一攻击合约中调用isSolved()
  9. 成功

Signin

目标

treasure.unlocked()

思路

  1. 在函数getHiddenKey()中，当我们满足sig1 == sig2,unlocked会被设为true,这个函数有两个约束条件

    1. msg.sender == owner，在函数becomeOwner()中，满足msg.value > treasureAmount以及msg.sender != tx.origin即可将调用者设为owner

      1. 可以使用一个攻击合约将msg.sender与tx.origin分开。Attack为msg.sender直接调用这个函数，EOA为tx.origin调用attack()
      2. 目标合约部署时，Setup将treasureAmount设为了1 ether,所以只要我们在实例化时传入2 ether,msg.value = 2 ether > treasureAmount，满足条件

    2. keccak256(bytes(_string1)) != keccak256(bytes(_string2))，即要求我们输入的两个字符串经过keccak256计算后的结果不一致，即传入的字符串不能一样。而

​ 又要求它们keccak256计算后前四个字节一致

​ 在solidity中，存在函数选择器，它是取calldata（对函数名与参数类型进行keccak256计算）的前四个字节。如果我们找到一组存在函数选择器碰撞的函数,把它们作为_string1 ,_string2传入，就能顺利通关

  1. 在Attack合约中写attack()函数,并设为payable（receive():接收EOA转账），先调用becomeOwner()并传入2 ether;调用getHiddenKey()，将找到的_string1 ,_string2传入
  2. 在脚本中新实例化Attack，调用attack()并传入2 ether
  3. 调用isSolved()
  4. 成功

Capital_prevails

目标

使address(proletariat)持有的rweb数量大于10 ether

思路

  1. 在Setup合约中，使用实例capita_prvails调用createTrade创建了一笔交易。对应Capital_prevails的struct Trade。想要卖出10 ether rweb,买入1 ether weth

  2. Capital_prevails合约中的三个函数，如下

    1. createTrade(),允许用户创建一个代币交换订单。

      1. 约束条件是：_tokenToSell不能为空地址，

      2. 并且要支付_amountToSell(safeCast(_amountToSell - fee)的_tokenToSell,其中fee为30wei

        maker ==函数调用者

        taker==成交者（此时设为address(0))

        amountToSell用户想要出售的数量-手续费(30 wei)

        failledAmountToSell和filledAmountToBuy初始化为0

        isActive为true表示订单有效

    2. scaleTrade()，这是一个扩大规模的函数

      1. 只能由交易的发起者[maker]进行调用

      2. 要求传入的scale >0

      3. 订单必须是isActive,trade.filledAmountToBuy == 0,

    初始amountToSell不足以支付手续费的时候，需要从msg.sender转账到合约

    1. settleTrade()，结算交易，允许任何用户来成交一个已有的订单

      tradeAmount = _amountToSettle * trade.amountToBuy,要求

      ​ trade.filledAmountToSell + _amountToSettle <= trade.amountToSell

      当我们向ID为_trade的交易转tradeAmount / trade.amountToSell后，对应交易会给我们转_amountToSettle。

      因为Setup中发起的订单是第一个订单，所以它的_tokenId为0，因为并未要求msg.sender必须是maker，所以我们可以直接调用。而需要我们转代币的公式

      ==tradeAmount / trade.amountToSell==存在精度损失问题。当我们传入一个极小的值，tradeAmount / trade.amountToSell会被向下取整为0，即我们不需要支付任何代币，即可获得_amountToSettle的rweb。（1 ether = 10^18^wei,必须传入整数，所以一次最多可以薅 9 wei)

      但每次调用我们必须保证trade.filledAmountToSell > trade.amountToSell

    2. cancelTrade()，取消交易如trade.amountToSell - trade.filledAmountToSell > 0，说明有剩余的amountToSell没有被settle，需要退回给maker

  3. 授权，允许Capital_prevails使用我们合约的代币

  4. 连续调用settleTrade获得部分rweb代币

    初始amountToSell是10ether - 30 wei ~ 10 ether，9*1e18/10e18=0.9向下取整，我们就不需要支付weth,但能免费获得90wei rweb

  5. 我们可以创建一个恶意交易，出售31wei的rewb ,购买0个weth,合约扣除30 wei手续费后，amountToSell = 1 wei,amountToBuy = 0

  6. 调用scaleTrade,将恶意交易放大，为了不再支付手续费，我们可以将scale设为一个极大的数（amountToSell = safeCast(safeMul(trade.amountToSell, scale))不能超过2^112^,此处传入一个近似于2^112^的数）,trade.amountToBuy == 0

    originalAmountToSell == 1,

    因为newAmountNeededWithFee在做safeCast时结果2^112^ >2^112^-1,发生溢出，为0。

    所以实际上，我们不用支付任何费用

  7. 调用settleTrade(),我们想要合约给我们转_amountToSettle的tokenToSell因为TokenToBuy被设置为0，所以我们无需支付手续费，就可将合约的rweb全部转出

  8. 利用ERC20中的函数transfer将rweb代币转给地址proletariat

  9. 调用isSolved()
